# コードレビュー

指定されたPR・ブランチの変更内容をレビューする。

対象: $ARGUMENTS

引数が空の場合は、現在のブランチとデフォルトブランチの差分を対象とし、その旨を報告してから開始する。

---

## 1. レビュー手順

### Step 1: 変更内容の把握

1. 対象の差分を取得する（`git diff`, `gh pr diff` 等）
2. 変更ファイル一覧と変更行数を確認する
3. 変更の目的・背景を把握する（PR説明文、コミットメッセージ）

### Step 2: 観点別レビュー

以下の観点で順番にチェックする。指摘は必ず該当ファイルを読み、`file:line` の根拠を確認してから報告する（差分のみからの推測指摘は禁止）:

#### 正確性（Must Check）
- [ ] 変更が意図した動作を実現しているか
- [ ] エッジケースの考慮漏れはないか
- [ ] null / undefined の安全な処理ができているか
- [ ] 既存機能を壊していないか

#### セキュリティ（Must Check）
- [ ] ユーザー入力のバリデーションは適切か
- [ ] 認証・認可のチェック漏れはないか
- [ ] 機密情報（トークン、パスワード等）がハードコードされていないか

#### 設計・保守性
- [ ] 責務の分離は適切か（1関数1責務）
- [ ] 不要な複雑さはないか（過度な抽象化、不要なユーティリティ）
- [ ] 既存のコードパターン・規約に従っているか
- [ ] 既存のディレクトリ構造に準拠しているか（ファイル配置・命名規則）
- [ ] 既存のコンポーネントやユーティリティを再利用しているか（車輪の再発明がないか）
- [ ] マジックナンバーや意味不明な文字列がないか

#### エラーハンドリング
- [ ] API通信のエラー処理は適切か（異常系のレスポンスハンドリング）
- [ ] try/catch の範囲は適切か（広すぎ・狭すぎがないか）
- [ ] エラー時のユーザーへのフィードバックがあるか（トースト、エラーメッセージ等）

#### SSoT（Single Source of Truth）
- [ ] 型定義が一箇所で管理されているか（不要なre-exportや重複定義がないか）
- [ ] 定数・設定値が一箇所で管理されているか
- [ ] 同じロジックが複数箇所に散在していないか

#### パフォーマンス
- [ ] 不要な再レンダリングの原因になっていないか（フロントエンド）
- [ ] N+1 クエリ等の非効率なDB操作がないか（バックエンド）
- [ ] 大量データ処理時のメモリ効率は適切か

#### 型・静的解析
- [ ] 型定義は正確か（any の使用、不必要な型アサーション）
- [ ] 未使用の import / 変数がないか

### Step 3: 指摘の再検証（報告前に必須）

Must Fix / Should Fix と判定した指摘は、報告前に Agent ツールのサブエージェントに反証を依頼する:

```
以下のレビュー指摘が本当に問題か、対象コードを読んで反証を試みてください。
- 指摘: <file:line> <指摘内容>
- 出力: 「正しい / 誤り / 条件付き」の判定と根拠
```

- **誤り** と判定された指摘は報告から除外する（誤指摘の防止）
- **条件付き** は根拠を添えて Suggestion に降格する

---

## 2. 指摘の重要度分類

| ランク | 定義 | 対応 |
|--------|------|------|
| **Must Fix** | バグ、セキュリティ問題、データ破損の可能性 | merge前に必ず修正 |
| **Should Fix** | 設計上の問題、保守性の低下、パフォーマンス懸念 | 強く推奨 |
| **Suggestion** | より良い書き方の提案、好みレベルの指摘 | 任意 |

---

## 3. 出力フォーマット

```markdown
## レビュー結果

### 概要
- 対象: PR#XXX / ブランチ名
- 変更ファイル数: N
- 変更行数: +X / -Y

### 指摘事項

| # | ファイル:行 | ランク | 内容 | 修正案 |
|---|-----------|--------|------|--------|
| 1 | ... | Must Fix | ... | ... |
| 2 | ... | Should Fix | ... | ... |
| 3 | ... | Suggestion | ... | ... |

記入例:
| 1 | src/api/user.ts:42 | Must Fix | id の null チェックなしで DB 照会しており、未認証時に 500 になる | 早期 return で 401 を返す |

### 良い点
- ...

### 総合判断
- [ ] merge可能 / [ ] 修正後にmerge可能 / [ ] 要再設計
```

---

## 4. デザイン照合レビュー

デザインURLが提供された場合は、通常のコードレビューに加えて以下も実施する:

- UI要素がデザインと一致しているか
- テキスト文言・ラベルの差異
- レイアウト・余白の差異

（詳細な照合手順は `/design-check` skill を参照）

---

## 5. スコープ制御

### やること
- 変更差分に対する体系的なレビュー
- 具体的な修正案の提示
- merge可否の判断

### やらないこと
- レビュー中に直接コードを修正する（指摘のみ。修正はユーザーの指示後）
- 変更差分に含まれないコードへの指摘（関連する場合は「参考」として言及可）
