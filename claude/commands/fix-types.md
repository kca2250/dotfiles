# 型エラー・静的解析エラーの修正

型チェックエラーや静的解析の警告を体系的に修正する。

対象: $ARGUMENTS

---

## 1. 手順

### Step 1: 現状把握

1. 型チェックコマンドを実行してエラー一覧を取得する
2. エラーを以下の基準で分類する:

| 分類 | 説明 | 例 |
|------|------|-----|
| **依存エラー** | 型定義元の変更に起因 | 共有スキーマの変更による型不整合 |
| **実装エラー** | コード自体の型誤り | 引数の型不一致、null安全性 |
| **未使用** | 未使用import/変数/型 | unused import, declared but never used |
| **SSoT違反** | 型の重複定義・不必要なre-export | 同じ型を複数箇所で定義 |

### Step 2: 依存関係順に修正

修正順序が重要。以下の順番で対応する:

1. **型定義ファイル** （source of truth）
2. **共有スキーマ・コントラクト**
3. **サービス・ロジック層**
4. **UIコンポーネント層**
5. **テストファイル**

理由: 上流のエラーを先に直さないと、下流で連鎖エラーが発生する。

### Step 3: 修正ごとの検証

1つのエラー群を修正するたびに:

- [ ] 型チェックを再実行して **新たなエラーが発生していない** ことを確認
- [ ] 修正が他のファイルに波及していないことを確認

---

## 2. 修正方針

### 修正してよいもの
- 明らかな型の不整合（引数型の修正、null チェックの追加等）
- 未使用の import / 変数の削除
- 単純な re-export の SSoT 準拠（不要な re-export の削除、import 元の統一）

### ユーザーに確認が必要なもの
- 型定義自体の変更（インターフェースの構造変更等）
- any / as による型アサーションの追加（根本解決ではないため）
- ロジック変更を伴う型修正

---

## 3. 完了条件

- [ ] 型チェックコマンドがエラー0で通過する
- [ ] 新たな `any` や型アサーション `as` を追加していない（やむを得ない場合は理由を報告）
- [ ] 修正内容の一覧をユーザーに報告済み

---

## 4. 完了時の報告

```

- 修正前エラー数: N件
- 修正後エラー数: 0件

### 修正内容

| # | ファイル | エラー内容 | 修正方法 |
|---|---------|-----------|---------|
| 1 | ... | ... | ... |

### 注意事項（あれば）
- ...
```
