# 実装計画の策定

指定された機能・修正について、実装計画を策定する。

対象: $ARGUMENTS

---

## 1. 計画策定の手順

### Step 1: インプットの収集

以下を確認・収集する:

- [ ] 仕様書（提供されている場合は読了）
- [ ] デザインデータ（提供されている場合は確認）
- [ ] 既存の関連実装（同種の機能がどう実装されているか）
- [ ] プロジェクトのディレクトリ構造・命名規則

### Step 2: 現状分析

- 既存コードの構造を把握する
- 変更が必要なファイル・モジュールを特定する
- 技術的な制約・依存関係を洗い出す

### Step 3: 計画の策定

以下のテンプレートに従って計画を作成する:

```markdown
## 実装計画: [機能名]

### 前提条件
- ...

### スコープ
- 含むもの: ...
- 含まないもの: ...

### ステップ

#### Step 1: [タイトル]
- 目的: ...
- 変更対象: ...
- 完了条件: ...

#### Step 2: [タイトル]
- 目的: ...
- 変更対象: ...
- 完了条件: ...

（続く）

### リスク・懸念事項
- ...

### 見送り事項（今回のスコープ外）
- ...
```

### Step 4: 自己精査

計画策定後、以下の観点で精査する:

- [ ] 既存コードとの整合性は取れているか
- [ ] 既存の型定義・定数・共通ロジックを把握し、再利用する前提になっているか（SSoT）
- [ ] ステップの依存関係は正しいか（順序を入れ替えても成立するか）
- [ ] 各ステップの粒度は適切か（1ステップ = 1つの論理的な変更単位）
- [ ] 抜け漏れているファイル・モジュールはないか
- [ ] 影響範囲の見落としはないか

---

## 2. ステップ設計のガイドライン

### 粒度の基準
- 1ステップ = 1コミット相当の変更量
- 1ステップ完了時点でアプリが壊れない状態を維持（段階的に動く状態を保つ）

### ステップの順序

#### 大幅なリファクタリング・修正の場合（テスト先行）
1. **一時テスト作成**（変更前の振る舞いを保証するテストを先に書く）
2. テストがグリーンであることを確認
3. 型定義・スキーマ（データの形を先に決める）
4. バックエンド（API・サービス層）
5. フロントエンド（UI・ロジック）
6. 一時テスト再実行（既存の振る舞いが壊れていないことを確認）
7. **一時テストファイルを削除**
8. クリーンアップ（不要コードの削除等）

#### 新規機能追加・軽微な修正の場合
1. 型定義・スキーマ（データの形を先に決める）
2. バックエンド（API・サービス層）
3. フロントエンド（UI・ロジック）
4. テスト
5. クリーンアップ（不要コードの削除等）

---

## 3. 実行時のルール

計画実行時は以下のルールに従う:

- **ステップごとに報告**: 何を変更し、何ができるようになったかを説明
- **ステップ間で確認**: 次のステップに進む前にユーザーの承認を得る
- **計画の修正**: 実装中に計画の変更が必要になった場合、変更内容と理由を説明してから修正

---

## 4. 出力

- 計画をユーザーに提示し、承認を得る
- ユーザーから指示があれば、**`.ai-notes/`** ディレクトリにmdファイルとして保存する（存在しない場合は作成する。保存後は相対パスを報告）
