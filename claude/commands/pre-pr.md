PR を出す前に、差分を 4 つの観点で自己レビューしてください。**全項目を機械的に列挙するのではなく、重要度（critical / major）に該当する指摘だけを報告**します。問題が見つからなければ「OK」とだけ返します。

## 対象リポジトリ

`~/dev/omron-pf-app`（Turborepo + pnpm workspace の Next.js モノレポ）

このスキルは上記リポジトリの構成（`apps/*`, `packages/*`, `docs/specs`, `docs/bdd` など）を前提に書かれています。別リポジトリに流用する場合は中身を読み替えてください。

## 進め方

1. **差分取得**
   - ベースブランチは引数で受け取る（既定: `develop`）。
   - `git diff <base>...HEAD --stat` でファイル一覧、`git diff <base>...HEAD` で詳細を取得。
   - 引数で対象パスが渡された場合はそのスコープに限定。
   - 差分が 0 行 / 巨大すぎる（>2000行）場合はその旨を伝えて中断。

2. **4 観点で並列に判定**（後述）
   - 各指摘に `[critical] / [major] / [minor]` のラベルを付ける。
   - **report する閾値: critical / major のみ**。minor は内部判定だけして、明示要求がない限り出さない。
   - critical / major と判定した指摘は、報告前に Agent ツールのサブエージェントで反証を試みる。反証が成立した指摘は除外し、判断が割れたものは根拠を添えて major → minor に降格する。

3. **報告フォーマット**
   ```
   ## 自己レビュー結果

   ### 🔴 critical (要修正)
   - <file:line> — <指摘内容> / <なぜ問題か>

   ### 🟡 major (要確認)
   - <file:line> — <指摘内容> / 確認ポイント

   ### ✅ 問題なし
   - 差分スコープ
   - 仕様準拠
   ```
   - 該当 0 件のセクションは省略可（全てクリアなら「すべての観点でクリア。PR 提出 OK」とだけ返す）。

4. **最後にユーザーに一行で確認**
   - critical があれば「先に修正しますか？」
   - major のみなら「このまま PR を出しますか？それとも確認しますか？」

## 重要度の定義

| ラベル | 基準 |
|--------|------|
| critical | merge するとバグ・障害・情報漏洩を引き起こす。型エラー、未処理の例外、シークレット混入、明らかな仕様逸脱、後方互換破壊 |
| major | 動くが PR としてマージ前に直すべき。テスト未追加、エッジケース未考慮、a11y 違反、パフォーマンスの明らかな悪化 |
| minor | スタイル・命名・コメントレベル。**この報告では出さない**（要求されたら出す） |

## 観点 1: 差分スコープと テスト追加

- PR の目的と関係ない変更が混ざっていないか（フォーマッタ差分、無関係パッケージの bump 等）
- 1 つの PR に詰め込みすぎていないか（>10ファイル かつ 異なるドメイン横断は major）
- 変更されたロジック（`apps/*`, `packages/*` の `.ts/.tsx`）に対し、対応する `*.test.ts(x)` が追加 / 更新されているか
- DB スキーマ変更（`packages/db/`）にマイグレーションとシードの整合があるか

## 観点 2: 破壊的変更・エッジケース・エラーハンドリング

- 公開 API（`packages/api`、tRPC ルーター、エクスポート関数）のシグネチャ・戻り値の破壊的変更 → critical
- React コンポーネントの props 削除 / 必須化 → 利用箇所の更新漏れがないか
- `null / undefined / 空配列 / 空文字` の扱い、`Promise.all` の中の失敗、try/catch で握りつぶしていないか
- ローディング / エラー / 空状態 の UI が考慮されているか（FE 変更時）
- DB クエリの N+1、トランザクション境界

## 観点 3: デッドコード・セキュリティ

- `console.log`, `debugger`, 大きなコメントアウトブロック、`TODO` 残し（追加 TODO は major）
- `.env`, 認証トークン、秘密鍵、ハードコードされた接続文字列のコミット → critical
- ユーザー入力の dangerouslySetInnerHTML / 未エスケープ表示
- `apps/*/next.config.*` の `images.domains` や CSP の緩和、CORS 設定の緩和
- 認可チェック（Cognito / セッション）の抜け


## 観点 4: 仕様準拠

- 変更内容に対応する仕様ドキュメントを `~/dev/omron-pf-app/docs/specs/`, `~/dev/omron-pf-app/docs/bdd/`, `~/dev/omron-pf-app/docs/architecture/`, 該当機能の README から探す。
- 見つかった仕様と差分を突き合わせ、**仕様と食い違う実装** または **仕様にない振る舞いの追加** を指摘。
- 仕様ドキュメントが該当変更について言及していない場合は「該当仕様ドキュメントなし」と明記（指摘ではない）。
- BDD の Given/When/Then と実装の振る舞いが食い違う → critical。

## 補足ルール

- **lint / typecheck / test は実行しない**。本スキルは差分の人間レビュー観点に特化（自動チェックは CI / 別スキルの責務）。
- 各指摘は **`file:line — 指摘 / なぜ` の 1 行**で簡潔に。長い説明は避ける。
- 推測ベースの指摘は出さない。差分と仕様から根拠が示せるものだけ。
- ユーザーが「全部見せて」と言った場合のみ minor も含めて報告する。
