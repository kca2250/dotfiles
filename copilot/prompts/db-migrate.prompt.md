# DB・スキーマ変更

DBスキーマの変更とマイグレーションを安全に実施する。

対象: $ARGUMENTS

---

## 1. 安全確認チェックリスト（着手前に必ず確認）

- [ ] 現在のスキーマファイルを読了したか
- [ ] 変更対象のモデル・テーブルを特定したか
- [ ] 既存データへの影響を評価したか（破壊的変更の有無）
- [ ] マイグレーション戦略を把握したか（push / migrate deploy / 手動SQL）

### 破壊的変更の判定

以下は **破壊的変更** であり、ユーザーへの確認が必須:

| 操作 | リスク |
|------|--------|
| カラム削除 | データ消失 |
| カラム名変更 | 既存クエリ・コードが壊れる |
| NOT NULL制約の追加（既存データあり） | マイグレーション失敗の可能性 |
| ユニーク制約の追加（重複データあり） | マイグレーション失敗の可能性 |
| テーブル削除 | データ消失 |

---

## 2. 実装手順

### Step 1: スキーマ変更

1. スキーマファイルを修正する
2. 変更内容を報告する（追加/変更/削除したフィールド一覧）

### Step 2: マイグレーション

ユーザーと確認の上、環境に応じた手順を取る:

- **ローカル開発**: `db push` 等の即時反映コマンド
- **ステージング/本番**: マイグレーションファイルの生成と適用

### Step 3: 関連コードの更新

スキーマ変更に伴い、以下を更新する:

- [ ] サービス層（CRUD操作）
- [ ] APIコントラクト / スキーマ定義
- [ ] フロントエンドの型定義
- [ ] シードデータ（該当する場合）
- [ ] テストデータ・テストコード

### Step 4: 整合性の検証

- [ ] 型チェックが通ること
- [ ] スキーマファイルとマイグレーションファイルが整合していること
- [ ] API側で新フィールドが正しくレスポンスに含まれること

---

## 3. トランザクション関連の注意事項

ORM のトランザクション内では:

- トランザクション変数（`tx` 等）を使用すること
- ORM クライアント（`prisma` 等）を直接使わないこと
- 既存コードで上記違反がある場合は報告する

---

## 4. スコープ制御

### やること
- スキーマ変更とそれに直接関連するコードの更新
- 破壊的変更のリスク報告

### やらないこと
- commit / push の自動実行
- 本番環境への直接的なマイグレーション実行
- スキーマ変更に関連しないコードの修正

---

## 5. 完了時の報告

```
## スキーマ変更結果

### 変更内容
| モデル/テーブル | 操作 | フィールド | 型 | 備考 |
|---------------|------|-----------|-----|------|
| ... | 追加/変更/削除 | ... | ... | ... |

### 更新ファイル一覧
- (相対パス)

### 破壊的変更
- なし / あり（詳細）

### 次のステップ
- マイグレーションコマンドの実行（ユーザーが実施）
```
