# 調査

指定された対象について構造的に調査し、結果を報告する。

対象: $ARGUMENTS

---

## 1. 調査の進め方

### Step 1: 調査スコープの確定

調査開始前に以下を明確にする:

- **何を調べるのか** （バグ原因 / 仕様との乖離 / 実装状態 / 依存関係 etc.）
- **調査範囲** （特定ファイル / 特定機能 / アプリ横断 etc.）
- **期待する出力** （原因特定 / 一覧リスト / 修正方針 etc.）

不明な場合はユーザーに確認する。

### Step 2: 仮説の立案

調査対象に基づいて仮説を立てる:

- 問題が発生している可能性のある箇所を推定
- 確認すべきファイル・コードパスをリストアップ

### Step 3: 証拠収集

- 関連ファイルの読み込み
- Grep / Glob でのパターン検索
- git log / git blame での変更履歴確認
- 必要に応じてサブエージェントで並列調査

### Step 4: 分析・結論

- 収集した証拠から仮説を検証
- 根本原因（Root Cause）を特定
- 影響範囲を評価

---

## 2. サブエージェント活用の判断基準

以下の場合はサブエージェントで並列調査する:

- 調査対象が **複数の独立した領域** にまたがる（FE/BE、複数アプリ等）
- 調査項目が **5件以上** あり、それぞれ独立して調べられる
- 仕様書ファイルが **複数** あり、それぞれ照合が必要

以下の場合はサブエージェントを使わない:

- 調査対象が1ファイル・1機能に限定される
- 調査項目間に依存関係がある（Aの結果がBの入力になる等）

---

## 3. 出力フォーマット

### 調査結果レポート

```markdown
## 調査概要
- 調査対象: ...
- 調査範囲: ...
- 調査日: YYYY-MM-DD

## 発見事項

| # | 分類 | 内容 | 重要度 | 対応方針 |
|---|------|------|--------|---------|
| 1 | ... | ... | High/Medium/Low | ... |

## 根本原因（該当する場合）
...

## 推奨対応
1. ...
2. ...

## 未調査事項（あれば）
- ...
```

### mdファイルへの記録

ユーザーから指示があった場合、調査結果をmdファイルに出力する。

- 出力先は **`.ai-notes/`** ディレクトリ（存在しない場合は作成する）
- ファイル作成後は **相対パスを必ず報告** する

---

## 4. スコープ制御

### やること
- 指定範囲内の網羅的な調査
- 発見事項の構造的な整理と報告
- 修正方針の提案

### やらないこと
- 調査中に見つけた問題の修正（報告のみ。修正はユーザーの指示後）
- 調査範囲外への拡大（必要と判断した場合はユーザーに提案して確認）
